# 核心概念速查

GitHub 的术语对设计师来说最大的障碍不是难,而是陌生。它们原本是为软件工程诞生的词汇,搬到设计场景往往显得别扭。下面这张对照表用设计师熟悉的概念来理解这些术语。

## 基础术语

| 术语 | 设计师的理解 |
|---|---|
| Repository(仓库) | 项目文件夹,含所有文件和完整的历史记录 |
| Branch(分支) | 同一项目的平行版本,就像 Sketch 文件的"另存为试验版" |
| Commit(提交) | 每次保存一个有意义的版本快照,附带一句说明 |
| Push | 把本地改动上传到远程仓库 |
| Pull | 把远程最新版本下载到本地 |
| Fork | 复制别人的项目到自己账号,在此基础上做自己的版本 |
| Clone | 把远程仓库完整下载到本地电脑 |
| Pull Request | 申请把自己的改动合并回原项目,简称 PR |
| Issue | 问题反馈或功能建议的留言板 |
| Star | 收藏项目,相当于 Dribbble 的点赞 |
| README | 项目首页说明文档,用 Markdown 写成 |
| Release | 正式发布的版本,通常附带可下载的打包文件 |

## 设计师场景下的重要概念

### Commit message(提交说明)

每次 commit 都要写一句话说明改了什么。好的习惯是把它当作"给未来的自己写便条"——三个月后回看,这句话应该能帮你准确回忆起当时的修改。

不推荐:`更新` / `修改` / `改了一下`

推荐:`重写首页介绍段落` / `补充字体推荐的权重信息` / `修复目录里的死链接`

### .gitignore

一个告诉 Git "这些文件不用管" 的清单文件。对设计师最有用的场景是屏蔽 macOS 的 `.DS_Store`、Adobe 的临时锁文件、以及各种过大的素材文件。

### GitHub Pages

GitHub 提供的免费静态网站托管服务。把一个仓库里的 Markdown 或 HTML 文件自动变成可访问的网页,URL 通常是 `用户名.github.io/仓库名`。**这是设计师最容易获得的第一个"作品上线"体验**。

## 心智模型:Git 到底在做什么

如果要用一句话解释,Git 就是一个**文件版本的时间机器**,它为每一次"有意义的保存"拍一张快照,所有快照按时间排成序列,你可以在任意时刻跳回任何快照。

但它有一个重要前提:**擅长处理文本文件,不擅长处理大型二进制文件**。

- 对 `.md` / `.txt` / `.html` / `.css` / `.svg` 源码 —— 能精确到字符的 diff 对比
- 对 `.docx` / `.psd` / `.ai` / `.indd` —— 只能记录"文件换过了",无法展示细节变化
- 对 `.tiff` / `.mp4` / `.raw` —— 会让仓库迅速膨胀,实践上不建议纳入

理解了这一点,就理解了为什么设计师不能简单照搬工程师的 Git 工作流——物理基础不同。
