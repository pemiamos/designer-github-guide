# 设计师的 GitHub 工作流思考

这一章是这份指南的核心,也是最有争议的部分。

大部分现有的"设计师 GitHub 入门"文章,都在尝试回答一个问题:**如何像工程师一样管理设计文件?** 但这个问题从根本上就提错了。

## 工程师的 Git 为什么能成立?

工程师的生产物是**文本**——代码、配置、文档。文本有几个关键属性让 Git 变成神器:

1. **可逐字 diff** —— 能精准看到"第 42 行从 A 改成了 B"
2. **轻量** —— 一个大型项目的代码库通常也就几十 MB
3. **线性可合并** —— 两个人改同一个文件,Git 能自动合并(大部分时候)
4. **无损压缩** —— 一个文件改 1000 次,仓库也只会稍微变大

## 设计师的生产物是什么?

设计师的生产物以**二进制加视觉**为主——PSD、AI、INDD、大图、视频。这些文件:

1. **不可 diff** —— 两个 PSD 文件你看不出差异,只能肉眼对比
2. **极重** —— 一个完整的出版项目几百 GB 起步
3. **不可合并** —— 两个设计师同时改一个 PSD,只能二选一
4. **无法压缩存储** —— 改 10 次版本,仓库就膨胀 10 倍

所以**"设计师直接照搬工程师的 Git 工作流"这条路走不通**。这不是设计师没学会,是物理基础决定的。

## 真正可行的三条路径

### 路径一:作品展示层

GitHub 作为作品集平台。具体形态包括:

- **GitHub Pages** —— 把作品集做成静态网站,免费托管,用 `用户名.github.io` 域名
- **Profile README** —— 个人页面可以像 LinkedIn 一样精心设计,放图片、视频、项目列表
- **Releases 当发布渠道** —— 做好的字体、图标集、设计系统打包发布,别人可以下载

这一层的价值是:**被搜索、被发现、被引用**。在做设计系统、字体、插件的独立设计师圈,GitHub 就是主场。

### 路径二:创作过程层(最被低估)

这条路径的核心洞察是:**设计师的工作产出,远不止视觉源文件**。

大量高价值产物其实是文本友好的:

| 设计产物 | 传统媒介 | Git 友好形态 |
|---|---|---|
| 设计规范文档 | PDF / Word | Markdown |
| 组件库说明 | Figma 内注释 | Markdown + HTML |
| 设计决策记录 | 通常不记 | Markdown(ADR) |
| 课程大纲与讲义 | PPT | Markdown |
| 用户研究记录 | Word / Notion | Markdown |
| 项目日志与复盘 | 微信聊天 | Markdown |
| 色板与设计 token | Sketch 内部 | JSON / CSS |
| 图标库 | AI 文件 | SVG(天然文本) |
| 原型交互描述 | Figma 评论 | Markdown |
| 交互式信息可视化 | — | HTML |

这些才是设计师应该放进 Git 的东西,它们不是"设计工作的副产品",**它们本身就是设计工作的核心产出**——只是传统观念不把它们当"设计品"。

### 路径三:创作品本身层(接受物理限制)

即"用 Git 管理 PSD、AI、INDD"这些源文件。坦白说,在 Git 当前的架构下走不通。折中方案有:

- **Git LFS** —— 让 Git 管理大文件,但本质是把大文件存在另一个服务器。免费额度有限,很快就需要付费
- **DVC** —— 机器学习社区发明的大文件版本控制,设计师圈用的人很少
- **Perforce / Plastic SCM** —— 游戏影视行业用的专业版本控制系统,能处理大二进制文件,但昂贵复杂

对大多数设计师的建议是:**这一层不要强求 Git,用 Time Machine 或云盘版本历史更合适**。不要试图让一个工具解决所有问题。

## 设计的"文档化转向"

看看 IBM Carbon、Material Design、Shopify Polaris——顶级设计系统的本质是**文档**,不是 Sketch 文件。设计师这个职业的生产重心,正在从"做图"悄悄转向"定义系统加写下规则"。

这个转向给 GitHub 对设计师的意义提供了新的诠释:它不是一个"强迫你学习工程师工具"的地方,而是一个**让你的设计思考能够被组织、被检索、被传承、被协作**的基础设施。

## AI 时代让这条路彻底贯通了

传统上"写文档"对设计师来说很痛苦——设计师想画,不想打字。但 AI 工具(Claude、Cursor、各类笔记插件)大幅降低了"把思考转化为结构化文本"的摩擦。

一段口头描述的设计评审,可以被 AI 自动整理成结构化的决策记录。一张手绘草图,可以被 AI 辅助转化为 SVG 图标的描述文件。一份课程讲义,可以边和 AI 对话边成型。

**"设计师像工程师一样工作"的技术障碍,正在被 AI 消除**。这个趋势刚刚开始。

## GitHub 对设计师的社交价值

Behance 和 Dribbble 像是设计师的"时尚杂志",GitHub 则像是设计师的"学术期刊"。前者靠视觉冲击力,后者靠**可复用性和技术深度**。

一个设计师在 GitHub 上发一套精心制作的、带详尽文档的图标库,比在 Dribbble 上发 100 张飞机稿更能建立长期专业声誉。因为 GitHub 的评价标准不是"好看",而是"有用"——这恰好是成熟设计师应该追求的。

## 一个结论

对设计师而言,融入 GitHub 不是一次性的技术学习,而是一次**生产方式的再定义**。

把你日常工作里那些"一直存在但从未被认真对待"的文本产物——设计决策、规范说明、项目日志、课程材料、原型文档——郑重地放进一个版本管理系统里。给它们赋予和视觉产物同等的地位。

这件事做下去,你会慢慢发现自己的专业积累开始以前所未有的方式生长。
