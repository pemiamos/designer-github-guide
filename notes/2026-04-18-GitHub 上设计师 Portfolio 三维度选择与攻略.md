# GitHub 上设计师 Portfolio 三维度选择与攻略

> 本文写给所有希望用 GitHub Pages 搭建个人作品集的设计师——无论你是工业设计、交互设计、视觉设计、信息设计还是建筑设计方向的学生或从业者。全文从**技术栈**、**视觉设计语言**、**内容架构与叙事**三个维度展开选择空间，并针对每个维度中最推荐的方向给出实操攻略。

---

## 为什么选 GitHub Pages 做作品集

在开始之前，先说清楚 GitHub Pages 对设计师的真正价值。

它的优势不是免费（这是副产品），而是**完全的视觉控制权**。不像 Cargo、Squarespace、Notion 那样受模板限制，不像 Behance、站酷那样混在千万张缩略图里，GitHub Pages 让你决定每一个像素、每一次交互、每一段排版。对于把"作品集本身也当作品来做"的设计师，这是唯一的选择。

它的代价是需要学一些代码。但在 AI 辅助编码成熟的今天，这个门槛已经大幅降低——你不需要成为程序员，只需要理解基本的结构和语法，剩下的可以靠 AI 协作完成。

## 一、技术栈的具体选择

### 全景介绍

这里按"静态站点生成器"这一层细分，因为部署、域名、CDN 这些基础设施层差异不大，真正影响日常工作流的是生成器的选择。

**Astro**
设计师友好度最高的现代选择。核心理念是"默认零 JavaScript"——页面加载时纯 HTML/CSS，只在需要交互的地方"孤岛式"注入 JS 组件。优势在于：每个项目页可以用不同技术组件（React、Vue、Svelte、原生 HTML）混搭，适合需要嵌入多种交互原型的作品集。MDX 支持让你可以在 Markdown 里直接嵌入交互组件。图片优化内置。学习曲线中等。

**Next.js（静态导出模式）**
React 生态的主流选择。如果你未来想做更复杂的交互应用（带数据库、用户登录），Next.js 的生态更成熟。但对纯展示型作品集来说，它偏重。已经熟悉 React 的人可以顺手选择；没有 React 基础的话，直接学 Astro 更划算。

**Jekyll**
GitHub Pages 原生支持，写 Markdown 就能生成网站。适合项目多、更新频繁、愿意接受统一模板的情况。学术圈的个人主页大量用这个。对设计师来说略显技术化，主题生态偏"开发者审美"。

**Eleventy (11ty)**
极简主义者的选择。没有框架约束，你给它什么模板它就用什么。优点是产出极快、极轻。缺点是组件化和交互性弱，不适合需要嵌入大量交互原型的场景。

**Nuxt / SvelteKit**
Vue 和 Svelte 生态的对应物。Svelte 的动效语法对设计师来说很优雅，但生态小，AI 辅助编码的训练数据相对少，遇到问题不如 React 生态好解决。

**纯 Vite + 手写**
最轻的方案。只用 Vite 做构建工具，页面结构自己写。如果想把作品集当作品来做、每个项目页完全独立设计，这是最自由的选择。但没有内容管理层，加项目要手动维护导航。

### ⭐ 推荐选择：Astro

### 进一步攻略

#### 1. 为什么 Astro 特别适合大多数设计师

- **混搭自由**：不同项目可以用不同的技术栈实现。比如一个项目用 D3.js 做数据可视化，另一个用 Three.js 做三维演示，再一个就是静态图文，都能在同一个站点里共存。
- **内容与代码分离**：Astro 的 Content Collections 让你用 Markdown/MDX 写项目叙述，用组件处理视觉呈现。这契合"先把内容想清楚，再决定视觉"的工作方式。
- **性能默认优秀**：零 JS 的策略意味着你的作品集在手机、弱网络环境下也快。潜在雇主或合作方在地铁上点开你的链接，不会白屏 3 秒。
- **AI 辅助友好**：Astro 的训练数据在主流 AI 编程助手里非常充分，使用 AI 辅助开发几乎不会卡壳。

#### 2. 核心技术栈组合建议

- **框架**：Astro（最新稳定版）
- **样式**：原生 CSS + CSS 变量（不要急着上 Tailwind。手写 CSS 能保持对视觉的完全掌控，Tailwind 会把你拉向"组件库审美"）
- **内容**：MDX（Markdown + JSX），让你能在写作时直接嵌入交互组件
- **图像**：Astro 内置的 `<Image>` 组件，自动生成 WebP/AVIF、多尺寸响应式
- **交互图表（按需）**：D3.js、Observable Plot、或你项目所需的专门库
- **字体**：Fontsource（npm 安装开源字体，避免 Google Fonts 的隐私和网络问题）
- **部署**：GitHub Actions 自动部署到 GitHub Pages
- **分析（可选）**：Plausible 或 Umami（比 Google Analytics 轻、无 cookie）

#### 3. 项目结构规划

```
portfolio/
├── src/
│   ├── content/
│   │   ├── config.ts              # 内容模型定义
│   │   ├── projects/              # 项目 .mdx 文件
│   │   ├── writings/              # 写作 .md 文件
│   │   └── themes/                # 主题介绍文本
│   ├── components/
│   │   ├── narrative/             # 叙事组件（引用、图注、分隔线等）
│   │   ├── viz/                   # 可视化组件（可嵌入的交互原型）
│   │   └── layout/                # 导航、页脚等
│   ├── layouts/
│   │   ├── BaseLayout.astro       # 站点通用骨架
│   │   ├── ProjectLayout.astro    # 项目页骨架
│   │   └── ThemeLayout.astro      # 主题页骨架
│   ├── pages/
│   │   ├── index.astro            # 首页
│   │   ├── themes/
│   │   ├── projects/[...slug].astro
│   │   ├── writing/
│   │   └── about.astro
│   └── styles/
│       ├── global.css             # 全局样式、CSS 变量
│       └── typography.css         # 排版系统
├── public/
│   ├── fonts/                     # 本地字体文件
│   └── images/                    # 原始高分辨率图
└── astro.config.mjs
```

#### 4. 学习路径（假设每周投入约 8 小时）

- **第 1 周**：跑通 Astro 官方教程，理解 Content Collections、Layout、路由机制
- **第 2 周**：搭好项目骨架和设计系统的 CSS 变量层，不填内容
- **第 3 周**：把一个代表性项目做成样板项目页
- **第 4 周**：抽象出可复用组件，做首页
- **第 5-6 周**：补齐其他页面，部署上线

如果你每周只能投入 3-4 小时，把周期乘以 2 就是比较真实的估计。

#### 5. 需要规避的坑

- **不要一开始就追求完美**：Astro 有极强的扩展性，容易陷入"还有更好的写法"的优化陷阱。先做出丑但能用的版本，上线后再迭代。
- **不要用太多第三方组件库**：shadcn、Radix UI 这些对开发者很方便，但会让你的作品集变得像 SaaS 产品。设计师的作品集应该是自己手写的。
- **图片管理要有纪律**：原图放 `public/images/`，用 Astro 的 `<Image>` 组件引用。不要直接用 `<img src>`，会失去自动优化。
- **MDX 文件不要超过 500 行**：一个项目页如果内容太长，拆成多个组件导入，否则写作时会很痛苦。

## 二、视觉设计语言的策略

### 全景介绍

这不是"选一个好看的风格"的问题，而是"用什么视觉策略来承载你的专业身份"的问题。常见的策略大致分四类：

**编辑设计型（Editorial）**
把作品集做成一本在线杂志。大量使用网格系统、严谨的字体层级、留白、经典印刷传统的版式语言。参考对象是 It's Nice That、Eye Magazine 网站、Kenya Hara 的个人网站。适合内容型设计师、有写作习惯的设计师。风险是容易做成"千篇一律的瑞士风格"，需要用内容的独特性来打破。

**画廊型（Gallery）**
视觉优先，文字极少。大图、全屏、悬停展开细节。适合以视觉成品为主的设计师（平面、插画、摄影、服装）。对工业/交互设计师来说，这种策略会丢失"思考过程"这个重要维度，不太推荐。

**档案型（Archive）**
反设计的设计。像一个整洁的文件系统或学术档案，强调信息密度和可检索性。传达"我把时间花在思考而非装饰"的气质。对研究型、学术倾向的设计师有天然契合度。

**实验型（Experimental）**
把网站本身做成一件交互作品。每次滚动、点击、悬停都是设计。Bruno Simon、Active Theory 那类。强烈表达能力，但维护成本极高，且容易让内容被形式淹没。适合交互设计师、创意编程方向的设计师做一个"签名项目"，不适合作为长期维护的作品集主调。

**混合型**
外壳是编辑/档案型的克制风格，但在项目页内部允许实验性的表达。这样导航和浏览体验是稳定可读的，而每个项目能用最适合它的视觉语言说话。

### ⭐ 推荐选择：混合型

**为什么对大多数设计师是最佳默认选择**：它避免了"风格统一但单调"和"每个项目很炫但站点混乱"两个极端，同时让你的作品集有长期延展性——未来加入新项目时，不会被早期定下的一套强视觉风格绑架。

### 进一步攻略

#### 1. 混合型的核心原则

**"外壳克制，内核自由"**——这是贯穿整个设计决策的口诀。

- **外壳**（首页、导航、主题页、列表页、关于页、页脚）：统一、克制、近乎家具化的设计。读者不应该"注意到"这些页面的设计，它们应该像书的装帧一样透明。
- **内核**（每个项目页的正文区域）：每个项目可以有独特的视觉语言，甚至独特的交互行为。但要与外壳在字体、基础间距等少数"锚点"上保持连续性，避免跳戏。

这种策略的精神前辈是高质量的印刷杂志：封面、目录、页眉页脚高度统一，但每篇专题文章的版式可以截然不同。

#### 2. 外壳的设计系统

**字体（两到三种足够）**

根据你的专业方向和气质选择搭配：

- **偏向编辑/人文气质**：中文用思源宋体或方正悠宋，英文用 EB Garamond、Crimson Pro 或 Source Serif 4
- **偏向现代/中性气质**：中文用思源黑体，英文用 Inter、Söhne、Neue Haas Grotesk
- **偏向技术/理性气质**：标题用无衬线，正文也用无衬线，但挑层级差异丰富的字族（如 Inter 的 Thin 到 Black）
- **等宽字体（可选）**：用于代码、图注、元数据。JetBrains Mono 或 IBM Plex Mono

同时使用超过三种字族几乎总是错的。

**尺度系统（模块化音阶）**

- 基础字号 16-18px，比例 1.25（Major Third）适合大多数场景
- 派生出完整层级：脚注、正文、引导段、小标题、章节标题、页面标题、Hero 标题
- 间距系统基于 8px：4、8、16、24、32、48、64、96、128

**颜色（极度克制）**

作品集阶段建议限制在四到五种颜色：

- 主背景（一个偏暖或偏冷的近白）
- 主文字（不用纯黑，用 #1A1A1A 或类似的深灰黑）
- 次级文字（图注、元数据用的中灰）
- 分隔线（极浅的中性色）
- 强调色（一种——只一种——有个性的颜色，用于链接、按钮、关键视觉锚点）

选强调色的一个技巧：**从你代表作项目的主色中抽取一种**，这样整个外壳的视觉温度会和你的作品自然连贯。颜色留给作品内部去丰富。

**栅格**

- 桌面端 12 列栅格，最大内容宽度约 1200-1280px
- 正文阅读区约 680-720px 宽（阅读舒适度的黄金区间）
- 全宽图像、交互原型可以打破栅格延伸到更宽区域

**动效**

外壳的动效应该几乎看不见：链接下划线的过渡、图片 fade-in、导航的微微位移。不要做视差滚动、鼠标跟随粒子、文字逐字浮现这些"设计师作品集标配"——它们已经成了新的俗套。

#### 3. 内核的自由区间

每个项目页可以在以下维度做独特设计：

- **排版节奏**：可以改变正文宽度、图文比例、页边距
- **色彩系统**：可以引入该项目特有的色板
- **交互行为**：滚动触发的动画、鼠标悬停显示的信息、可拖动的控件
- **章节分隔**：可以用该项目特有的视觉元素作为章节间的分隔装饰

但要保持的"锚点"（这些保持统一，避免读者产生跳到另一个网站的感觉）：

- 顶部导航条（可以淡化但不能消失）
- 项目元数据区（年份、媒介、合作者）的排版风格
- 基础字体（即使改变大小和颜色，字族保持一致）
- 脚注和引用的样式

#### 4. 建立你的视觉语言参考库

建议建一个 Obsidian 笔记、Notion 数据库或 Pinterest 板专门收集参考。按层级整理：

- **外壳参考**：你希望整个站点的"底色"看起来像什么。杂志设计、书籍装帧、优秀的博客都是好来源。
- **项目页参考**：沉浸式长报道、交互叙事、好的作品案例研究。New York Times、The Pudding、FiveThirtyEight、日本的 NHK 特集网页都是宝库。
- **同类设计师参考**：找 10-20 个你尊敬的同类设计师的作品集，记录他们做对和做错的地方。
- **反例库**：记录你觉得"设计过度"、"套路化"、"喧宾夺主"的作品集，提醒自己不要走向那里。

#### 5. 避免的陷阱

- **不要追逐流行**：Glassmorphism、Neumorphism、Brutalism 这些样式都会过时。编辑型的克制美学是穿越时间的。
- **不要让"设计"喧宾夺主**：如果读者看完页面先想到"这个网站做得真酷"，你输了；应该是先想到"这个项目真有意思"。
- **不要过度动画**：每一个动画都应该回答"它帮读者理解了什么"。装饰性动画只会增加认知负担。
- **深色模式不是必须**：如果你没有精力为深色模式单独调整所有图片、图表、颜色对比度，就暂时不做。勉强做的深色模式只会显得业余。

## 三、内容架构和项目页的叙事方式

### 全景介绍

这是最容易被忽视但最决定作品集成败的层面。技术和视觉都只是容器，叙事才是内容本身。

#### 站点级架构的常见逻辑

**按项目时间倒序**——最常规，但隐含"越新越重要"的假设，对长期作品不公平。

**按项目类型分类**——"产品设计 / 交互设计 / 视觉设计 / 教学"。清晰但僵硬，交叉领域的作品难归类。

**按主题或探究方向**——这是研究型设计师的优势路径。用类似"材料的表达"、"日常物的重构"、"老年人的数字体验"这样的主题作为顶级分类，每个主题下挂相关项目、写作、实验。这种组织方式立刻把你和普通设计师拉开距离——它展示的不是作品清单，而是一个持续的研究议程。

**扁平时间线 + 强过滤**——所有内容（项目、文章、讲座、实验）混在一起按时间排，但提供多维度标签过滤。适合内容产出频繁、跨类型的创作者。

#### 项目页级叙事的常见结构

**问题-过程-结果（经典案例研究）**——最标准的结构，但也最容易写得无聊。八股化的陷阱就在这里。

**决定性时刻驱动**——不按时间线走，而是围绕项目中 3-5 个关键决策点展开：为什么选择这个切入角度？为什么放弃某个方向？哪个洞察改变了整个设计？这种结构尊重读者的时间，突出思考而非流程。

**成果先行，过程可折叠**——开头直接展示最终成品（图、视频、可交互原型），读者被勾住后，再往下提供过程细节作为"为什么这样做"的回答。符合大多数人的浏览行为。

**沉浸式长文叙事**——把项目页当成一篇有设计感的长文来做，图文交错、滚动触发、小标题引导。适合研究性强、有故事性的项目。

**对话/访谈式**——用自问自答或假想对话来展开。罕见但记忆点强，对有教学身份的人有契合度。

### ⭐ 推荐选择
- **站点级架构**：按主题或探究方向
- **项目页级叙事**：成果先行，过程可折叠

**为什么这个组合适合大多数认真做作品集的设计师**：主题式架构让你的作品集从"作品列表"升级为"研究陈述"——这是学生和普通从业者最容易拉开差距的地方；"成果先行"的叙事则尊重了读者的浏览习惯，既能留住快速扫视的读者，也能为深度读者提供思考过程。

### 进一步攻略

#### 1. 主题式架构的设计原则

**主题是你的"研究方向"，不是作品的分类标签**

一个好的主题应该满足：

- 是一个**可以持续追问的问题**，而不是已完成的领域。"日常物的情感连接"是主题；"产品设计"不是——它只是一个领域标签。
- **能跨越媒介**。每个主题下应该能容纳项目、文章、教学、实验、未完成的草稿。
- **数量在 3-5 个之间**。少于 3 个显得单薄；多于 5 个就失去了聚焦感。
- **主题之间可以有交叉**。允许一个项目同时归属两个主题，这恰恰展示了你思考的立体性。

#### 2. 如何找到你的主题

如果你是刚毕业的学生或工作不久的设计师，你可能会觉得"我没有什么研究方向"。但实际上你只是没有**把它显影出来**。

试试这个方法：

1. 列出你过去 2-3 年做过的所有项目（包括课程作业、竞赛、实习、自主项目）
2. 对每个项目问：**它真正让我兴奋的是什么？不是成果本身，而是其中哪个问题？**
3. 把这些"让你兴奋的问题"写成小纸条，摊开在桌上
4. 找出它们之间的模式——哪些纸条可以归为一组？这些组就是你的主题雏形

举几个例子，帮助你理解什么样的表述是"主题"：

- "被忽略的日常物的重新设计" ✓
- "人与公共空间的非预期互动" ✓
- "传统工艺在当代生活中的转译" ✓
- "为边缘群体设计的界面" ✓
- "品牌设计" ✗（这是领域）
- "我的 UI 作品集" ✗（这是类型分类）

如果你现在只有 5-6 个项目，可能只能支撑 2 个主题——这完全没问题。主题会随着你的作品积累逐渐清晰。

#### 3. 主题页应该包含什么

主题页不是简单的"项目列表"。它应该是一篇短小的**研究陈述**，让读者理解这个主题对你意味着什么：

- **开篇引言**（200-400 字）：这个主题的核心问题是什么？你为什么关心它？你在这个方向上已经走到哪里？
- **关键作品**（3-6 个）：最能代表这个方向的项目卡片，配简短说明
- **相关写作**：属于这个主题的文章/笔记列表
- **教学延伸（如果有）**：你在课堂中如何展开这个主题，或你为谁分享过这个方向
- **当前思考（可选）**：最近在追问什么问题、下一步想做什么。这是最能让读者"感受到你还活着"的部分。

这种结构让主题页本身成为一篇值得读的内容，而不是导航节点。

#### 4. 项目如何关联主题

在 Astro 的 Content Collections 里，给每个项目加 `themes` 字段（允许多选）：

```yaml
---
title: 项目标题
themes: [theme-slug-1, theme-slug-2]
year: 2024
medium: 产品设计 / 交互原型
---
```

这样一个项目可以出现在多个主题页里，展示你思考的交叉性。

#### 5. "成果先行，过程可折叠"的项目页结构

**第一屏：Hero 成果展示**
- 全屏视觉。一张震撼的成品图，或一个直接可交互的原型片段。
- 极少量文字：项目标题、一句核心陈述（不是摘要，是这个项目的"论断"）。
- 滚动提示（很细微，比如一个向下的小箭头或"继续阅读"的链接）。

**第二屏：项目元数据与核心问题**
- 左侧：年份、媒介、合作者、发表渠道等元数据
- 右侧：一段 150-300 字的核心陈述，说清楚这个项目是什么、它回答了什么问题、你的角色是什么
- 这是全项目最重要的一段文字。读者如果只看到这里就离开，也要带走一个清晰的印象。

**第三段：关键成果展示（3-6 个片段）**
- 用精心挑选的成品图、交互片段、成品细节，按视觉冲击力降序排列
- 每个成果配 1-2 句点睛的说明，不要展开过程
- 这一段的目标是让读者"看完就懂这个项目的价值"

**分隔线：进入过程区域的信号**
- 一条视觉上明确的分界。可以是留白、一行小标题"设计过程 / Process"、或一个可点击的"展开过程"按钮。
- 这个分隔告诉读者：以上是成果，以下是思考。如果你只想看成果，到这里可以离开了。

**过程区域（可折叠或长页面滚动）**
- 这里展开关键决策点。推荐用"决定性时刻"而非时间线。
- 每个决策点有独立的小标题（H3 级别）和 200-500 字说明，配相关的过程材料（草图、参考、中间版本、反面案例）
- 典型的决策点举例：
  - 为什么选择这个切入角度？
  - 哪个约束条件改变了设计方向？
  - 哪个方案被放弃了，为什么？
  - 哪个技术/媒介的限制变成了机会？
  - 哪个用户反馈让你重新思考？
- 3-6 个决策点足够。不要写成完整的项目日志。

**尾声**
- 一段简短的反思：这个项目让你对你的主题有了什么新理解？
- 相关链接：该项目所属的主题页、相关的其他项目、如果有发表的文章或出版物
- 致谢（如有合作者）

#### 6. "可折叠"的三种实现方式

- **纯滚动式**：不真的折叠，就是一个长页面，但用视觉分隔明确成果区和过程区。读者可以自然停在任何位置。优点：简单、SEO 友好、打印友好。**推荐首选**。
- **按钮展开式**：过程区默认折叠，需要点击"展开设计过程"才出现。优点：页面更短，不想看过程的读者没有滚动负担。缺点：多一次交互。
- **侧边锚点式**：页面左/右侧有小目录，读者可以跳转到感兴趣的决策点。优点：过程区内容多时更好导航。适合特别长的项目页。

对于沉浸式叙事的重点项目，**纯滚动式 + 侧边锚点**是最好的组合。

#### 7. 写作项目页的工作纪律

- **先写，后设计**：不要在 Figma 里排版时构思内容。先用 Markdown 把全文写出来，自己通读一遍觉得能打动人，再进入视觉布局。
- **成果区和过程区分别能独立成立**：一个好的测试——把过程区全部删掉，成果区是否还是一个完整的项目展示？应该是。这证明你的成果本身有说服力。
- **避免"设计过程八股"**：不要写"首先我做了用户调研，然后绘制了用户画像，接着制作了线框图..."。这种流水账没人想看。只写那些**让这个项目成为现在这样**的决定。
- **每个决策点都要有"反面"**：说清楚"我选了 A，而不是 B，因为..."。没有 B 的 A，读起来就是结果描述；有 B 的 A，才是决策。
- **控制长度**：整个项目页文字量建议在 1500-3000 字之间。少于 1500 字显得单薄；多于 3000 字读者会流失。复杂项目可以到 3500 字左右，但需要很好的视觉节奏。

## 动手前的推进顺序

三个维度的方向如果都按推荐选择定下来，下一步按这个顺序推进：

1. **确定 2-4 个主题的最终表述**——这决定了整个站点的骨架和潜台词。
2. **写出"关于"页的第一版**——这会强迫你把自己的定位想清楚，所有其他页面的调性都会跟着明确。
3. **启动 Astro 项目骨架**，搭建设计系统的 CSS 变量层（字体、颜色、间距）。
4. **选一个代表性项目作为样板**，从内容写作开始，而不是从代码开始。先把这个项目页做到接近完成度。
5. **样板完成后反向抽象组件**，再做首页和主题页。
6. **最后补齐外围页面**——写作列表、教学、关于详情等。
7. **部署上线**——不要等到"完美"再上线。60 分上线后迭代，远胜于 90 分永远躺在本地。

---

## 结语：作品集不是终点，是媒介

最后想说的是：作品集的价值不在于它多精美，而在于它能否**准确地传达你是谁、你关心什么、你思考的质量**。

一个视觉粗糙但内容真诚的作品集，永远胜过一个精致但空洞的作品集。技术栈和视觉策略只是让你能更清晰地表达——真正的功课是去想清楚**你到底要表达什么**。

而这个过程，本身就是一次对自己设计身份的重新设计。
