# 2026-04-18 · 建立这个仓库

## 动机

一直在断断续续地用 GitHub——浏览、下载、Star 收藏,但从未把它当作自己的日常工作基础设施。最近在研究如何为《长白山天池》这本科普书做版本管理,初步尝试中发现:**直接把 Adobe 源文件塞进 Git 是条死路**。

但在和 AI 的对话中理清了一件事:设计师进入 GitHub 的真正入口,不是管理设计源文件,而是把**设计师本来就应该产出的文本类材料**——文档、规范、笔记、决策记录、课程讲义、HTML 原型——郑重地交给版本管理。

于是决定先做这件事:建立一个公开的、关于"设计师视角 GitHub 指南"的仓库。

## 为什么这个项目适合作为第一个 GitHub 实战

- 纯文本内容,没有大文件,完美契合 Git 的能力
- 零手工维护负担——改一次 Markdown,commit 一次就结束
- 内容本身有公开价值,可以积累社区影响力
- 做完之后就完整跑过了一遍 GitHub 工作流,有了直观认知
- 和我的设计教育者身份天然契合

## 初始结构

```
designer-github-guide/
├── README.md                  项目首页
├── docs/                      正式文档
│   ├── 01-核心概念.md
│   ├── 02-使用方式.md
│   └── 03-设计师工作流.md
├── notes/                     学习笔记(本文所在处)
├── resources/                 资源推荐
│   └── 优质仓库推荐.md
├── assets/                    图片、截图等附件
└── .gitignore
```

## 今日收获

第一次完整地从"建仓库"到"第一次 commit"跑通全流程。原以为会更复杂,实际上关键阻力不在技术层面,而在**心理层面**——设计师下意识会觉得 GitHub 是"不属于自己的地方"。一旦跨过这道门槛,后面就只是在积累而已。
