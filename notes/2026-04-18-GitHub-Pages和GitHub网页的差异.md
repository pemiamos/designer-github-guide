# GitHub Pages 和 GitHub 网页的差异

> 一次"文件夹 404"踩坑引出的系统性认知。  
> 记录日期:2026-04-18

## 问题现场

在完成仓库搭建、开启 GitHub Pages、删除了 `notes/README.md` 之后,访问:

```
https://pemiamos.github.io/designer-github-guide/notes/
```

得到的是一个大大的 **404 File not found** 页面,而不是预期的文件列表。

但同时,访问同一个文件夹在 GitHub 网页上的地址:

```
https://github.com/pemiamos/designer-github-guide/tree/main/notes
```

一切正常,能看到文件夹里的所有 md 文件。

两个看起来差不多的 URL,一个正常一个 404。这里到底发生了什么?

## 核心结论

**GitHub 网页(github.com)和 GitHub Pages(github.io)是两套不同的系统**,它们面向不同的用途,对"没有 index 文件的文件夹"处理方式完全不同:

| 访问方式 | URL 形式 | 对空文件夹的处理 |
|---|---|---|
| GitHub 网页 | `github.com/用户名/仓库名/...` | 自动列出文件夹内所有文件 ✅ |
| GitHub Pages | `用户名.github.io/仓库名/...` | **必须有 index.html 或 README.md,否则 404** ❌ |

## 为什么会这样

### GitHub 网页的定位

GitHub 网页本质上是一个**代码托管平台的浏览界面**。它的设计目标是让开发者能够:

- 查看仓库的文件结构
- 浏览每个文件的内容
- 跟踪代码的修改历史
- 进行代码审查和协作

在这个场景里,"自动列出文件夹内容"是**必备功能**——就像 Finder 或文件资源管理器一样自然。所以 GitHub 网页会在你访问任何文件夹时,友好地列出里面的所有文件。

### GitHub Pages 的定位

GitHub Pages 本质上是一个**静态网站托管服务**。它的设计目标是让你把仓库里的 HTML/CSS/JS 直接部署成一个可访问的网站。

既然是"网站",它就遵循**普通网站的规则**。普通网站访问 `example.com/folder/` 时,服务器会去找这个文件夹下的 `index.html`(或配置的默认文件)返回给浏览器——如果找不到,就返回 404。

GitHub Pages 在这一点上更宽容了一些,它会依次查找:

1. `index.html`
2. `index.md`
3. `README.md`

找到其中任何一个,就把它作为"文件夹的首页"渲染出来。**三个都没有,就是 404**。

这不是 GitHub Pages 的 bug,这是**所有静态网站的通用规则**。

## 为什么这很容易让人困惑

这个差异有两个让人容易混淆的特点:

**1. URL 看起来差不多**

`github.com/pemiamos/xxx` 和 `pemiamos.github.io/xxx` 只差一个"位置",但背后是完全不同的服务。

**2. 在仓库内部浏览时,GitHub 网页的"自动列文件"太自然了,让人误以为这是 GitHub 的"默认行为"**

实际上它只是**GitHub 网页这个浏览工具**的特性。一旦内容被发布为"网站"(Pages),它就要遵守网站的规则,不再享有这个便利。

换句话说:**GitHub 网页是给开发者看的"文件管理器",GitHub Pages 是给访问者看的"网站"**。前者可以没有首页,后者必须有。

## 解决方案

### 方案一:给每个要 Pages 访问的文件夹配一个 README.md(最简单)

在每个需要从 Pages 访问的文件夹里放一个 `README.md`,内容可以:

- 简单介绍这个文件夹是做什么的
- 列出关键文件的链接(手工维护,但频率不高)
- 或者直接引导访问者去 GitHub 仓库浏览完整列表

这是**最符合 GitHub 生态习惯**的做法——很多开源项目都这么做。

### 方案二:做一个真正的 index.html

如果你对 Pages 页面的外观有要求,可以在文件夹里写一个 `index.html`,用 HTML 自定义这个"目录页"的样式。

这种方式能给访问者更好的视觉体验,但需要你会写 HTML/CSS,而且失去了 Markdown 的便利性。

### 方案三:关闭这个文件夹的 Pages 访问

如果某个文件夹**不需要对外发布**(只是给自己做归档用),可以用 Jekyll 的配置把它从 Pages 构建中排除,这样就不存在 404 问题——但访问者也无法通过 Pages 访问。

具体做法是在仓库根目录建一个 `_config.yml` 文件:

```yaml
exclude:
  - notes/
```

这会让 `notes/` 文件夹不被 Pages 处理,Pages 用户完全访问不到,但 GitHub 仓库浏览依然正常。

适合"我只想让 Pages 展示主要内容,次要内容留在仓库里就行"的场景。

## 设计师应该如何选择

对大多数设计师来说,**方案一(简单 README)**是最合适的。原因:

1. 写一次,长期有效
2. 符合 GitHub 社区习惯,访问者不会感到别扭
3. 不需要学 HTML 或 Jekyll 配置
4. 可以让 README 本身成为文件夹的"欢迎页",而不仅仅是目录

## 更深的观察:"自动化"有边界

这次踩坑让我意识到一件事:在追求"零手工维护"的过程中,容易走得太极端。

"文件夹自动列出所有文件"看起来是一个自然的诉求,但它在 Pages 这种"真正的网站"环境里**不是理所当然的**。有些情况下,**写一个简短的固定 README 比追求完全自动化更实际**——因为它一次完成,不需要更新,对访问者也更友好。

真正的"自动化",不是消灭所有手工操作,而是**让必要的手工操作尽可能少、尽可能不重复**。一个写一次就不用再动的 README,本质上也是自动化。

## 给其他设计师的建议

如果你也在用 GitHub Pages 托管自己的设计内容或作品集,以下几点值得注意:

| 建议 | 说明 |
|---|---|
| 为每个公开访问的文件夹配备 README | 避免 404,提升访问者体验 |
| 区分"GitHub 网页能做什么"和"Pages 能做什么" | 前者是浏览工具,后者是真网站,规则不同 |
| 不要过度追求"零 README" | 一个简短的固定 README 是最经济的方案 |
| 仓库根目录的 README 尤其重要 | 它同时是 GitHub 页面的首页和 Pages 网站的首页 |

## 小结

| 维度 | GitHub 网页 | GitHub Pages |
|---|---|---|
| 本质 | 代码托管平台的浏览界面 | 静态网站托管服务 |
| 文件夹空访问 | 自动列文件 | 需要 index.html 或 README.md |
| 主要用户 | 开发者、合作者 | 访问者、读者 |
| URL | github.com/... | 用户名.github.io/... |
| 对文件列表的依赖 | 内置功能 | 需要自己实现 |

理解了这一点,就能避免大多数 Pages 相关的 404 问题。
