# Wiki 自动化处理设置教程

← [返回 Wiki 主页](./📚%20Wiki%20主页.md)

> 本文记录从零到全自动的完整配置过程。核心目标：用 Web Clipper 保存文章后，无需任何手动操作，AI 自动完成分类、整理、写入 Wiki、更新日志。

---

## 一、系统架构

```
Web Clipper 保存文章
        ↓
  Clippings/ 出现新 .md 文件
        ↓
  clipping_watcher.py（watchdog 监听）
        ↓
  等待 3 秒（文件完全写入）
        ↓
  Phase 1：调用本地 Ollama（gemma4:26b）
  └── 输出 JSON：价值等级 / 目标知识库 / 日志文字 / 标签
        ↓
  ┌── A 级 → Phase 2：生成知识库条目 → 写入对应知识库文件
  ├── B 级 → 只更新学习日志
  └── C 级 → 学习日志记「已存档」
        ↓
  标记原文件 frontmatter（processed: true）
        ↓
  移入 RAW/ 文件夹（保留原文备查）
```

**涉及文件：**

| 文件 | 说明 |
|------|------|
| `clipping_watcher.py` | 核心执行脚本 |
| `~/Library/LaunchAgents/com.amosamir.clipping-watcher.plist` | macOS 开机自启动配置 |
| `~/clipping_watcher.log` | 运行日志 |

## 四、处理逻辑详解

### 4.1 价值等级判断（Phase 1）

脚本调用 Ollama，让模型以 JSON 格式输出：

| 字段 | 说明 |
|------|------|
| `value_level` | A / B / C |
| `destination` | 目标知识库名称 |
| `log_entry_text` | 学习日志的一句话描述 |
| `tags` | 从固定标签集里选择 |
| `reason` | 判断理由（仅记录日志） |

**A 级**：有独特洞察，进知识库 + 学习日志  
**B 级**：有用但无需展开，只记学习日志  
**C 级**：重复/低价值，学习日志记「已存档」

### 4.2 内容生成（Phase 2，仅 A 级）

根据目标类型生成不同格式：

- **知识库-xxx**：`## 标题 <a id="anchor">` + 一句话 + 核心内容 + 我的判断
- **提示词模板库**：`### 名称` + 适用场景 + 代码块（提示词原文）
- **Skills库**：`## Skill名称` + 一句话 + 使用方法 + 判断

### 4.3 去重保护

- 检查文件 frontmatter 中是否有 `processed: true`，有则跳过
- 用 `threading.Lock` 防止同一文件被并发处理两次
- 只监听 `on_created` 事件，不监听 `on_moved`（避免 Web Clipper 重命名临时文件时重复触发）

### 5.4 设置开机自启动（LaunchAgent）

在终端运行（整段复制粘贴执行）：

```bash
cat > ~/Library/LaunchAgents/com.amosamir.clipping-watcher.plist << 'EOF'
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.amosamir.clipping-watcher</string>

    <key>ProgramArguments</key>
    <array>
        <string>/usr/bin/python3</string>
        <string>/Users/amos/Library/Mobile Documents/iCloud~md~obsidian/Documents/Claude/clipping_watcher.py</string>
    </array>

    <key>RunAtLoad</key>
    <true/>

    <key>KeepAlive</key>
    <true/>

    <key>StandardOutPath</key>
    <string>/Users/amos/clipping_watcher.log</string>

    <key>StandardErrorPath</key>
    <string>/Users/amos/clipping_watcher.log</string>

    <key>EnvironmentVariables</key>
    <dict>
        <key>PATH</key>
        <string>/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin</string>
    </dict>
</dict>
</plist>
EOF
```

激活：
```bash
launchctl load ~/Library/LaunchAgents/com.amosamir.clipping-watcher.plist
```

验证：
```bash
launchctl list | grep clipping
# 看到 com.amosamir.clipping-watcher 及 PID 数字即为成功
```

## 七、常见问题

**Q：Ollama 超时 / 没反应**

gemma4:26b 是 26B 参数模型，处理长文章可能需要 2～3 分钟。如频繁超时，在脚本顶部增大 `TIMEOUT` 值：
```python
TIMEOUT = 300  # 改为 5 分钟
```

**Q：模型输出的 JSON 格式不对，Phase 1 解析失败**

查看日志里的原始回复，若模型总输出乱码可尝试切换更小的模型：
```bash
ollama pull qwen2.5:14b
# 然后修改脚本顶部 MODEL = "qwen2.5:14b"
```

**Q：文件处理了但没有移到 RAW**

查看日志里是否有 `Frontmatter 标记失败` 或 `移动文件失败`。最常见原因是文件名含特殊字符（如双引号），移动失败不影响内容写入，可手动把文件拖进 RAW 文件夹。

**Q：修改了脚本后如何生效**

```bash
launchctl unload ~/Library/LaunchAgents/com.amosamir.clipping-watcher.plist
launchctl load   ~/Library/LaunchAgents/com.amosamir.clipping-watcher.plist
```

**Q：想换模型**

编辑脚本顶部 `MODEL` 变量：
```python
MODEL = "你的模型名"   # ollama list 查看可用模型
```
改完重启服务即可。
